1. 서브모듈
 1.1 저장 용량
  - 깃 저장소에 용량한계가 없음
  - 저장소 크기가 크면 컴퓨터에 부담 => 통상적으로 1G 제공하고
 1.2 저장소 분리
  - 저장소 하나를 작은 저장소 몇 개로 분할할 수 있음
  - 서브 모듈 : 저장소 하나가 다른 깃 저장소를 포함하는 형태
   => 서브 모듈을 이용해 대형 프로젝트도 효과적으로 관리
 1.3 상하 관계
  - 저장소 간 상하 관계 
   => 부모 저장소와 자식 저장소 형태로 분리
 1.4 서브모듈 추가
  - 2개 이상인 저장소를 부모 자식 관계로 연결하기
  - git submodule -help : 명령어 확인
 1.5 저장소 연결
  - git submodule add 원격저장소URL 폴더이름
  - 서브모듈로 저장소가 추가되면 메인 저장소는 서브 저장소를 서브 폴더 형태로 취급
 1.6 설정 파일
  - 처음으로 메인 저장소에 서브모듈을 등록하면 깃은 루트 위치에 설정 파일을 생성
  - 설정파일이름 : .gitmodules
 1.7 모듈 커밋
  - 메인 저장소와 서브모듈인 자식 저장소 간 관계를 지속적으로 유지하려면 추가된 정보들을 계속 가지고 있어야함
  - 메인 저장소가 자식들의 정보를 계속 가지려면 이를 커밋해 저장해야함

2. 서브모듈 작업
  2.1 모듈 저장소
   - 서브 폴더는 서브모듈로 분리한 독립된 깃 저장소
  2.2 모듈 커밋
   - 서브 폴더에서 commit 시 자식의 원격 저장소를 갱신하는 것과 동일

3. 자식 저장소 갱신
  3.1 자식 저장소
   - parent 밑의 child 와 gitstudy11_child의 commit 정보가 갖지 않음
    => 자식 저장소에 연결된 원격 저장소를 이용해 실제 자식 저장소를 최신 커밋 정보로 갱신해야함
  3.2 자식 저장소 갱신
   - git pull origin master를 통해 커밋 내려받기 수행 => pull 명령어로 실제 자식 저장소를 최신 커밋 정보로 갱신
  3.3 자식 저장소 작업
   - 실제 자식 저장소의 코드를 수정해 자식 저장소의 원격 저장소로 전송
  3.4 부모 저장소 적용
   - 실제 로컬 자식 저장소에서 commit,push -> 로컬 부모 저장소에서 pull 하면 부모 저장소에 반영
  3.5 부모 저장소 갱신
   - 서브모듈을 갱신하면 메인 저장소는 자신의 서브 폴더가 변경된 것을 인식
    => 메인 저장소는 서브모듈의 변경 내용을 모니터링하고 있음 

4. 부모 저장소 복제
  4.1 부모 저장소 복제
   - 서브모듈의 저장소를 복제하거나 배포할 때는 부모/자식 관계도 같이 복제해야함
  4.2 모듈 업데이트
   - 메인 저장소를 복제할 때 서브모듈 정보만 같이 복제할 뿐, 실제 하위 저장소는 같이 복제하지 않음
    => 서브모듈의 하위 저장소는 직접 명령어를 실행해 가져와야함
   - git submodule init -- 서브모듈 초기화
   - git submodule update -- 서브모듈 업데이트
