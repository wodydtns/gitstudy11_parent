1. 깃에 새 파일 등록
 1.1 스테이지에 등록
  - 등록 : 워킹 디렉토리에 있는 파일을 스테이지 영역으로 복사하는 것
   => 실제 복사가 아닌 논리적인 기록을 변경하는 과정
  - unstage 상태의 파일들은 커밋할 수 없음 
  - 스테이지 영역에 등록하지 않은 파일은 커밋 작업에 미포함
 1.2 파일 등록 취소
  - tracked -> untracked 상태로 변경하는 것
  - git rm / git reset(한번이라도 커밋 시 사용) 사용
   ex) git rm index.html
       git rm --cached index.html => cached <스테이지 영역에서만 등록된 파일 삭제>
  - 커밋 후 삭제는 파일이 삭제 또는 변화된 것으로 간주 => 커밋된 파일은 리셋으로 삭제 후 정리해주어야함
  1.3 등록된 파일 이름이 변경되었을 때
   - 파일이름 변경하기 : git mv 파일이름 새파일이름
     => 과정 설명([git mv 파일이름 새파일이름] 과 동일한 결과)
       -> 1.mv 파일이름 새파일이름
       -> 2. git rm 파일이름
       -> 3. git add 새파일이름
2. 첫 번째 커밋
 2.1 HEAD
  - 정의 : 커밋을 가리키는 묵시적 참조 포인터
  - 최정적인 커밋 작업의 위치를 가리킴
  - 부모 커밋과 동일함
  - 최소한 한 번 이상의 커밋을 해야함 HEAD가 존재 
  - HEAD는 커밋될 때마다 한 단계씩 이동
 2.2 스냅샷
  - 파일에서 변경된 부분을 찾아 수정된 내용만 저장 
  - HEAD가 가리키는 커밋을 기반으로 사진을 찍음
  - 깃은 스냅샷 방식을 이용해 빠르게 버전의 차이점을 처리하고, 용량을 적게 사용
 2.3 파일 상태와 커밋
  - git commit -help : commit 옵션 보기
  - 깃 커밋은 HEAD와 스테이지 영역 간 차이를 비교해 새로운 객체 생성
  - 커밋 시 생성된 객체를 기록하는 것고 ㅏ동시에 구별할 수 있는 메시지를 같이 작성해야함(커밋 메시지)
  - 커밋은 파일 이름을 여러 개 사용하지 않고 하나만 가짐
    -> 메시지 문자열로 각 변경 객체들을 구분함 => 모든 커밋은 반드시 커밋 메시지를 작성해야한다
  - git commit -a : -a 옵션은 커밋을 하기 전에 자동으로 모든 ㅍ ㅏ일을 등록하는 과정을 미리 수행 
    => 파일 등록과 커밋을 동시에 실행
