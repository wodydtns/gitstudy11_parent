1. 브랜치 작업
 - 브랜치 : 저장 공간 하나에서 가상의 또 다른 저장 공간을 만드는 것 / 프로젝트를 독립적으로 관리
 1.1 깃 브랜치 특징
  * 가상 폴더
   - 가상 폴더 생성 / 외부적으로는 물리적인 파일 하나만 있는 것으로 보임
  * 독립적인 동작 
   - 브랜치를 이용하면 원본 폴더와 분리해 독립적으로 개발 작업 수행
  * 빠른 동작
   - 다른 버전 관리 도구보다 가볍고, 브랜치 전환이 빠름
   - Blob 개념을 도입해 내부를 구조화
   - Blob은 포인트와 유사한 객체 => 브랜치 변경 시 포인터를 이동해 빠르게 전환
   - 브랜치 명령 사용 시 내부적으로 커밋을 하나 생성해 브랜치로 할당

2. 브랜치 생성
  - 브랜치는 공통된 커밋을 가리키는 지점 / 커밋처럼 SHA1 해시키를 가리킴 -> 외우기 힘드므로 별칭 사용
  => 브랜치를 생성한다는 것은 기존 브랜치 또는 커밋에 새로운 견결 고리를 하나 더 만드는 것
  => 새 브랜치를 생성하면 포인터만 있는 브랜치가 생성
  - branch는 개발 분기점
 2.1 브랜치 생성
  - git branch 브랜치이름 커밋ID - 커밋ID는 마지막 커밋ID를 기준으로 브랜치를 생성 및 추가
 2.2 브랜치 이름
  - 작성 규칙
   * 기호(-)로 시작할 수 없음
   * 마침표로 시작할 수 없음
   * 연속적인 마침표로 시작할 수 없음
   * 빈칸, 공백 문자, 물결, ^, ? , *, [] 등을 포함할 수 없음
   * 아스키 문자 포함할 수 없음
   * 브랜치 이름 중복 불가

3. 브랜치 확인
 - git branch : 현재 브랜치 목록 확인
 - git rev-parse 브랜치이름 : 현재 브랜치가 어떤 커밋 해시 값을 가리키는 지 확인
   => 브랜치의 해시 값과 브랜치를 생성한 기준 커밋의 해시 값이 동일함
 - git branch -v : 브랜치 세부 사항 확인          

4. 브랜치 이동
 4.1 체크아웃
  - git checkout 브랜치이름 : 새로운 브랜치로 이동 
   => 워킹 디렉토리에 커밋하지 않은 내용이 있다면 브랜치를 변경할 수 없음
  - git checkout -- 파일이름 : 파일로 체크아웃
 4.2 브랜치 동작 원리
  - 브랜치가 이동하면 HEAD 포인터도 함께 이동
  - 브랜치를 변경하려면 기존 브랜치의 워킹 디렉토리를 정리해야함
 4.3 이전 브랜치
  - git checkout - : 이전 디렉토리로 이동(linux에서 가능)

5. 브랜치 공간
 5.1 브랜치 로그
  - git log --graph --all : 로그 출력 시 브랜치 흐름도 같이 보기
  - git show-branch --more=10 : more(출력될 커밋 개수 제한) 

6. 마지막 커밋
 6.1 상대적 위치
  - 깃의 HEAD 포인터는 다양한 명령어를 입력할 때도 기준점으로 사용
  - 상태적 커밋 위치를 지정할 때는 ^ , ~ 기호를 같이 사용
  - ex) HEAD^3 | HEAD~3
 6.2 AHEAD,BHEAD
  - AHEAD : 서버로 전송되지 않은 로컬 커밋이 있는 것 
            로컬 저장소의 HEAD 포인터를 기준으로 로컬 브랜치에 있는 커밋이 서버의 커밋 개수보다 많은 경우
  - BHEAD : 로컬 저장소로 내려받지 않은 커밋이 있는 것
            원격 저장소의 커밋이 로컬 저장소보다 더 최신 상태 
 6.3 자동 이동 옵션
  - git checkout -b 브랜치이름 : 브랜치 생성 + 체크아웃
 6.4 커밋 이동
  - 브랜치 이름 === 커밋 해시키
  - 브랜치 이동 시 커밋 해시키를 사용해 체크아웃 할 수 있음         
 6.5 HEAD를 이용한 이동
  - git checkout HEAD-[num]     
   ex) git checkout HEAD-1
  - git checkout - :이전 브랜치로 이동
 6.6 리모트 브랜치
  - 리모트 브랜치 : 원격 저장소에 생성한 브랜치
  - 원격 저장소와 연결된 로컬 저장소에서 새로운 브랜치를 생성해도 자동으로 원격 저장소에도 브랜치가 생성되는 것은 아님
  - 원격 저장소에 등록된 브랜치가 자동으로 로컬 저장소를 만드는 것도 아님
   => 별도의 명령으로 저장소를 동기화해야함
 6.7 브랜치 추적
  - 브랜치 추적(드래킹 브랜치) : 원격 저장소의 브랜치를 가리키는 것 / 이 포인터 정보는 .git/refs에 저장
  - 리모트 브랜치는 서버 간에 통신을 하고 나서 생성
  - 로컬 저장소의 브랜치를 원격 저장소에 동기화하려면 푸시 작업을 해야함
  - git push 원격저장소별치 브랜치이름 
   => push한 브랜치이름 개수 만큼 브랜치 개수 생성
 6.8 이름이 다른 브랜치
  - 로컬 브랜치와 리모트 브랜치를 수동으로 지정해 연결할 수 있음
  - git push origin 브랜치이름:새로운브랜치
 6.9 업스트림 트래킹
  - 정의 : 로컬 저장소의 브랜치와 원격 저장소의 브랜치가 업로드할 수 있도록 매칭되어 있는 것
  - 트래킹 브랜치(업스트림)는 리모트 브랜치와 로컬 브랜치를 연결해 주는 중간 다리 역할
